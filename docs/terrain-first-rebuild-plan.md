# Terrain-First Rebuild Plan — Malazan Stylized Google Earth

## Problem statement

The previous public prototype failed the actual product goal. It looked like either:

1. a flat PNG with overlays, or
2. a decorative rectangle/cube board with arbitrary miniatures.

The intended product is neither. The target is a **stylized Google Earth for the Malazan world**: explorable, terrain-first, source-faithful, attractive, and built from actual map-derived geography.

## Non-negotiable north star

- The World of Malazan map is the coordinate/source authority.
- Source pixel space is canonical: `10000 × 5571`.
- Every pin, river, mountain, forest, coastline, biome, basin, and label must be expressed in that coordinate space.
- Stylization is generated from traced/derived geographic data; it must not be guessed or randomly placed.
- The user should be able to zoom/fly/pan like Google Earth: world → continent → region → city.
- It should feel cute/cool/premium, but never toy-slop.

## Current authoritative inputs

- Full source mosaic: `../site-captures/worldofmalazan/worldofmalazan-z6-mosaic.png`
  - dimensions: `10000 × 5571`
- Source map metadata: `../site-captures/worldofmalazan/worldofmalazan-map-data.json`
  - Leaflet `CRS.Simple`
  - source-pixel coordinate system
- Location points: `public/data/locations.json`
  - 602 extracted locations

## What we stop doing

- No cube blanket as terrain.
- No decorative assets placed by vibes.
- No hiding bad geography under clouds/DOF/bloom.
- No calling a PNG overlay the product.
- No public deploys unless they represent the correct direction.

## Correct architecture

### Stage 1 — source-map atlas foundation

Build a true atlas foundation:

- image pyramid / tile pyramid from the 10k source mosaic,
- zoomable map canvas/WebGL viewport,
- exact source-pixel coordinate transform,
- search and 602 location pins in exact coordinates,
- no decorative 3D layer by default.

Deliverable: a clean, fast, zoomable source-faithful atlas.

### Stage 2 — cartographic extraction

Create structured data layers from the source map:

```ts
type SourcePoint = [number, number]; // x,y in 10000x5571 source pixels

type RiverFeature = {
  id: string;
  name?: string;
  rank: 1 | 2 | 3 | 4;
  certainty: 'confirmed' | 'inferred' | 'draft';
  source?: string;
  mouth?: SourcePoint;
  basinId?: string;
  points: SourcePoint[];
};

type AreaFeature = {
  id: string;
  name: string;
  kind: 'land' | 'water' | 'forest' | 'mountain' | 'desert' | 'plain' | 'swamp' | 'ice' | 'basin' | 'political';
  certainty: 'confirmed' | 'inferred' | 'draft';
  rings: SourcePoint[][];
};

type TerrainControl = {
  id: string;
  kind: 'ridge' | 'peak' | 'valley' | 'coast' | 'riverbed' | 'plateau';
  weight: number;
  points: SourcePoint[];
};
```

Required layers:

- coasts / land mask,
- oceans/seas/lakes,
- rivers,
- drainage basins,
- mountain ridgelines and peaks,
- forests,
- deserts/plains/swamps/ice,
- region/political boundaries,
- settlements/labels.

Deliverable: visible, inspectable vector data with authoring tools.

### Stage 3 — terrain synthesis

Generate a real terrain surface from layers:

- land/ocean mask controls base elevation,
- coasts produce shore gradients,
- mountain ridges raise terrain with falloff,
- rivers carve valleys and visible river channels,
- lakes/seas are flat water planes,
- forests/biomes control materials and vegetation density,
- cities are subtle anchored markers, not random toy stacks.

Deliverable: a continuous terrain mesh/tile set that can be explored.

### Stage 4 — stylized rendering

Only after geography is correct:

- painterly terrain materials,
- low-poly/miniature stylization from real features,
- rivers as carved blue ribbons/inlaid water,
- mountain ranges as coherent ridges,
- forests as clustered canopy fields following polygons,
- atmospheric clouds and shadows,
- DOF/cinematic focus as optional presentation mode.

Deliverable: cute-but-cool premium atlas.

## Runtime design options to evaluate

Sub-agents are researching this, but likely candidates:

- MapLibre/OpenLayers/deck.gl for source/tile/overlay foundation.
- Three.js/R3F terrain layer for stylized 3D.
- Static image tiles generated by script for GitHub Pages/Molotan hosting.
- GeoJSON-like source-pixel coordinates, not lat/lon.
- Web workers for vector simplification/tiling.
- Quadtree/LOD terrain tiles for performance.

## First implementation milestone after research

1. Generate source tile pyramid from z6 map.
2. Replace current public app shell with a clean zoomable atlas viewport.
3. Keep exact 602 location search/pins.
4. Add visible traced river layer scaffold with real authoring, not decoration.
5. Begin automated/semi-automated extraction of coast/water/mountain/forest masks.
6. Commit only after visual QA confirms it no longer looks like a PNG panel or cube board.

## Acceptance criteria for next public deploy

- The first screen reads as an explorable map/terrain atlas, not a board/card demo.
- Zoom and pan feel like map exploration.
- Location coordinates match the source map.
- Rivers/terrain layers are clearly explained as traced/derived/draft.
- No arbitrary cube terrain.
- No random stylized assets in default view.
- Public build must be better than the withdrawn version.
